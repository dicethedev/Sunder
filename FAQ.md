# Sunder — FAQ

> Answers to questions from developers, protocol teams, and hackathon judges.

---

## The Basics

**What is Sunder?**

Sunder is a self-hosted API service that lets you sign messages using threshold cryptography — meaning the private key is split across multiple nodes and never exists in full anywhere. You call a single HTTP endpoint, get a signature back, and no individual server ever held enough information to forge that signature on its own.

**What problem does it actually solve?**

Every protocol that moves value needs to sign things — bridge transfers, oracle attestations, DAO treasury withdrawals. Today most teams use either a single hot wallet (one breach = everything gone) or on-chain multisig (slow, expensive, visible). 

Sunder is the third option: off-chain threshold signing where the key never exists in full, at a fraction of the cost of hardware security modules.

**How is this different from a regular multisig?**

With multisig, each signer holds a complete private key and signs independently. One compromised machine exposes one complete key. With threshold signing, no machine ever holds a complete key — only a mathematical share of one. The shares are useless individually. Even if an attacker compromises T-1 nodes simultaneously, they learn nothing about the signing key.

**How is this different from HashiCorp Vault?**

Vault's Transit Secrets Engine is the closest analogy. The difference is architectural: Vault reassembles its master key in memory during operation. A compromised Vault server at the right moment exposes the key. Sunder never reassembles. The full key doesn't exist during signing — only partial signatures from independent nodes are combined. That's a fundamentally stronger threat model.

## The Cryptography

**Did you implement the threshold cryptography yourself?**

No and that's intentional. The cryptography comes from [Thetacrypt](https://github.com/cryptobern/thetacrypt), a production-quality threshold cryptography library developed by researchers at IC3 (the Initiative for Cryptocurrencies and Contracts). Sunder is the service layer on top of it: the HTTP API, deployment infrastructure, key management, and developer tooling that turns a research library into something a team can ship.

**Which threshold scheme does Sunder use?**

BLS04 on the BLS12-381 curve for the MVP. BLS04 is a pairing-based threshold signature scheme — it produces short signatures, supports efficient aggregation, and is well-studied. The BLS12-381 curve is the same one used by Ethereum's consensus layer, which makes on-chain verification straightforward.

**What is the threshold exactly?**

The default cluster is 3-of-5: five nodes each hold a key share, and any three of them can cooperate to produce a valid signature. The threshold T and total nodes N are configurable at setup. The system tolerates up to N-T node failures without losing signing capability.

**Can the key ever be reconstructed?**

Not through Sunder's signing path. The `assemble()` function in Thetacrypt combines partial signatures — not key shares. The mathematical operation happens in the exponent of the elliptic curve, meaning valid signatures are produced without ever computing the underlying secret. This is the core property of threshold signature schemes as distinct from threshold secret sharing.

**What does "the key never exists" actually mean?**

During `thetacli keygen`, a trusted dealer generates key shares using Shamir's secret sharing over the scalar field of the curve. Each share is distributed to one node. The dealer process terminates. The full key was never written to disk or transmitted — it existed briefly in the dealer's memory during setup and nowhere else. From that point forward, the only operations that occur are partial signing operations on individual shares.

## The Trust Model

**What are the current trust assumptions?**

Three honest ones worth stating clearly:

1. **The aggregator is trusted.** In v0.1, the aggregator coordinates signing and assembles the final signature. A compromised aggregator cannot forge signatures (it doesn't hold key shares) but could return incorrect results or drop requests. Byzantine-fault-tolerant aggregation is a roadmap item.

2. **The setup dealer is trusted.** Key shares are generated by a trusted dealer using `thetacli keygen`. The dealer sees all shares momentarily during generation. Distributed Key Generation (DKG) — where no single party ever sees all shares — is not yet implemented. The Thetacrypt researchers confirmed this is a known gap.

3. **Channels are authenticated but not encrypted.** Signing messages between nodes are not sensitive and don't require encryption. Key shares are distributed offline at setup, not over the network. TLS between nodes is recommended for production deployments.

**Is the trusted dealer setup a blocker for production use?**

For a proof of concept and many real-world deployments: no. The trusted dealer runs once, offline, in a controlled environment. After setup it's gone. This is the same model used by many production threshold systems today. For the highest-security environments, DKG removes this assumption entirely — and it's explicitly on the Sunder roadmap.

**What happens if the aggregator goes down?**

Requests fail until it recovers. The aggregator holds no key material, so there's no security impact — just availability. Multiple aggregator instances behind a load balancer is the production pattern, and it works because the aggregator is stateless between requests.

**What happens if T or more nodes are compromised?**

If an attacker controls T or more nodes simultaneously, they can produce valid signatures for arbitrary messages. This is the fundamental security bound of any T-of-N threshold scheme. The mitigations are operational: geographic distribution of nodes, independent infrastructure providers, proactive share refresh (on the roadmap), and monitoring for anomalous signing activity in the audit log.

## Using Sunder

**How do I integrate Sunder into my application?**

Two lines of Rust using the SDK:

```rust
let client = SunderClient::new("http://your-aggregator:8080");
let result = client.sign("bridge-signer", b"approve_withdrawal_4821").await?;
```

Or via HTTP from any language:

```bash
curl -X POST http://your-aggregator:8080/v1/sign/bridge-signer \
  -H "Content-Type: application/json" \
  -d '{"message": "617070726f76655f7472616e73666572"}'
```

**How do I run a cluster?**

```bash
# 1. Generate key shares (one time)
./scripts/setup.sh

# 2. Start 5 nodes + aggregator
cd docker && docker compose up
```

That's it. The full cluster is running.

**Can I use Sunder with an EVM chain?**

Yes. BLS04 on BLS12-381 is directly verifiable on Ethereum using the `BLS12_G2ADD` and related precompiles introduced in EIP-2537. A `SunderVerifier.sol` contract that accepts a Sunder-produced BLS signature is on the immediate roadmap.

**What message format does Sunder expect?**

Hex-encoded raw bytes. Sunder is message-agnostic — it signs whatever bytes you give it. The application is responsible for constructing and interpreting the message. Common patterns: `keccak256(abi.encode(params))` for EVM use cases, raw transaction bytes for signing transactions, or arbitrary application-defined payloads.

**How do I rotate keys?**

In v0.1: run `thetacli keygen` again with `--new` to generate a fresh set of shares, distribute to nodes, and restart. Proactive share refresh — rotating shares without changing the public key — is on the roadmap and is the production-grade answer to this question.

## For Hackathon Judges

**What did you actually build vs. what came from Thetacrypt?**

Thetacrypt provides: threshold signature math, key serialization, partial sign, assemble, verify.

Sunder provides: HTTP API layer, multi-node fan-out and coordination, named key management, API authentication, append-only audit logging, Docker deployment, Rust SDK, CLI, and all operator tooling. Everything above the dashed line in the architecture diagram.

**Why is the service layer valuable if the crypto already exists?**

Because Thetacrypt is a research library with no deployment story, no HTTP interface, no key naming, no audit trail, and no developer SDK. Before Sunder, using it required understanding the internals well enough to call its Rust types directly and managing node communication yourself. The researchers confirmed in conversation that the "biggest missing piece between Thetacrypt as a library and something a team could actually ship" is exactly the layer Sunder builds.

**What's the path from hackathon project to real product?**

The self-hosted open-source version is the developer onboarding experience. The commercial product is a hosted version: Sunder operates the node cluster across geographically distributed infrastructure, teams call the API and pay per signing request — the same model Infura uses for RPC, or AWS KMS uses for key operations. The total addressable market is every crypto protocol team doing key management, which is all of them.

**What would you build next with more time?**

In priority order: on-chain BLS signature verifier contract, TLS between nodes during key setup, proactive share refresh (rotate shares without changing public key), Byzantine-fault-tolerant aggregation, and eventually Distributed Key Generation to eliminate the trusted dealer assumption entirely — the gap the Thetacrypt researchers explicitly called out as the most interesting unsolved problem.
